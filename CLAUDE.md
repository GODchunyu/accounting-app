# CLAUDE.md - 记账 APP Agent 指南

本文件是本仓库 AI coding agent 的唯一权威启动约束。进入本仓库后，必须先读本文件，再读 `docs/PRD.md` 和 `docs/ROADMAP.md` 中与当前任务相关的章节。

不要新增另一套 `AGENT.md` / `AGENTS.md` 规则。`AGENTS.md` 只能作为 bootstrap 指向本文件，不能复制独立规则。

## 1. 文档分层

- `CLAUDE.md`：规定 agent 怎么工作，包括流程、测试纪律、文档回写、完成定义、git/PR 规则、业务铁律 A1-A8。
- `docs/PRD.md`：规定产品做什么，包括页面范围、用户角色、核心流程、数据模型、API 范围、验收标准。
- `docs/ROADMAP.md`：规定按什么顺序做，包括 Phase、任务编号 R-###、依赖关系、任务状态、ADR 架构决策记录。

凡引用业务铁律，一律使用本文件第 4 节的 A 编号，例如 A1 金额精度、A2 数据隔离。

## 2. 项目定位

本项目是个人记账 APP 的 H5 高保真原型，后端 Docker 化部署。产品核心闭环是：

```text
注册登录 -> 创建/选择账本 -> 记一笔 -> 查看明细 -> 查看统计 -> 管理分类/账本
```

第一版只做个人记账核心能力，不做预算、发现页、VIP、积分、徽章、邀请、家庭账单、资产、发票、优惠券、房贷、导入导出、OCR、支付方式、标签、周期账单等扩展功能。

所有开发都必须围绕核心记账闭环倒推，不允许为了实现方便偏离 PRD 和 ROADMAP。

## 3. 技术架构

目标架构以 `docs/ROADMAP.md` 为准，未更新文档前不得擅自引入第二套主要框架或绕过既定契约。

- 前端：React + Vite + TypeScript + 自定义 CSS + lucide-react。
- 后端：Node.js + Express + TypeScript。
- ORM / 迁移：Prisma。
- 数据库：PostgreSQL。
- 认证：用户名密码 + bcrypt + JWT。
- 图片上传：后端本地 `uploads` 目录 + Docker volume 持久化。
- 部署：Docker Compose。
- Monorepo：
  - `apps/web`：H5 前端。
  - `apps/api`：后端 API。
  - `packages/shared`：共享类型、常量、分类配置。
  - `docs`：权威文档。

## 4. 业务铁律 A1-A8

这些是不允许违反的不变式，等同 PRD 的核心约束。

- **A1 金额精度**：金额一律使用 Decimal / Prisma Decimal / 数据库 numeric，禁止用 float 计算金额。
- **A2 用户数据隔离**：用户只能访问自己的账本、分类、账单和图片。所有业务接口必须以后端鉴权结果为准，禁止信任前端传入的 userId。
- **A3 账本保护**：用户注册后必须有默认账本；不允许删除最后一个账本；删除账本必须二次确认，并删除该账本下账单及图片凭证。
- **A4 分类历史稳定**：已被账单使用的分类不得物理删除，只允许停用；历史账单必须继续显示原分类。
- **A5 图片凭证约束**：单笔账单最多一张图片，最大 5MB，仅支持 jpg/jpeg/png/webp；删除账单或更换图片时必须清理旧图片。
- **A6 账单归属一致**：账单的 userId、bookId、categoryId 必须属于同一用户；账单 type 必须与分类 type 一致。
- **A7 核心页面范围**：第一版只保留明细、图表、记账、我的四个核心页面，不出现已取消功能入口。
- **A8 认证底线**：密码必须 bcrypt 哈希存储；业务接口必须 JWT 鉴权；错误响应不得暴露敏感堆栈或密钥。

## 5. 权威文档同步

`docs/PRD.md` 和 `docs/ROADMAP.md` 是开发的上游和回写目标。每做一个功能、修复或架构调整，都必须检查这两个文档是否需要反向更新。

强制检查点：

1. 开始前：确认当前任务来自 ROADMAP 哪个 Phase / R-###；若不是，说明它映射到哪个 PRD 目标，并先补进 ROADMAP。
2. 写计划时：引用对应 ROADMAP 任务、PRD 章节、验收标准、相关业务铁律 A#。
3. 实现后：检查实际行为、接口契约、限制和测试是否改变 PRD 或 ROADMAP。
4. 完成前：若进度推进，更新 ROADMAP 对应任务状态；若产品定义或架构决策变化，更新 PRD 或 ADR。
5. 若无需更新，也要在最终回复或 PR 描述里说明已检查且无需回写。

不允许只改代码不回看文档，也不允许让文档与实现长期分叉。

## 6. 任务入口协议

开始任务前先判断任务类型：

- 用户要求诊断、定位、review、先不要改：只读代码、日志、PRD、ROADMAP 和测试，不做文件修改，直到用户明确要求执行。
- 用户要求实现、修复、补文档、提交：按本文件流程执行，不停留在建议层。
- 用户没有指定任务来源：从 ROADMAP 选择最高优先级、未完成、依赖已就绪的任务，优先 Phase 0 再 Phase 1。
- 用户指定任务但 ROADMAP 没有：先说明映射到哪个 PRD 目标，必要时补进 ROADMAP 后再开发。

所有 plan 必须先列验收标准和测试策略，再列实现步骤和 review/回写步骤。测试策略要说明哪些行为用单元测试、哪些协作用集成测试、哪些关键路径才用 E2E。

## 7. 标准开发流程

所有功能开发遵守以下流程，脚手架或纯文档小改可按改动规模缩短，但硬业务逻辑必须完整执行：

1. 从 ROADMAP 认领当前最高优先级任务。
2. 阅读 PRD 中相关产品、页面、数据模型、API、验收章节。
3. 明确验收标准：正常路径、异常路径、边界条件、可观察结果，并对应到 PRD 或 A#。
4. 生成执行 plan，第一部分必须是测试策略。
5. 按 TDD 写自动化测试，优先写最小、最快、最靠近业务规则的测试。
6. 运行新增测试，确认它因目标功能缺失而失败，不能是语法、数据或测试写错。
7. 写最小实现让测试通过。
8. 在测试通过前提下重构，不借重构新增未测试行为。
9. 自查 review：契约、边界、错误处理、权限隔离、金额精度、图片清理、文档回写。
10. 跑提交门禁：前端 lint/typecheck/test/build，后端 lint/typecheck/test，受影响关键路径跑对应集成或 E2E。
11. 若发现 bug，先补可复现失败测试，再修复，再 review，再重跑相关测试。
12. 重复 test -> implement -> review -> test -> fix，直到测试通过且证据完整。
13. 反向更新 ROADMAP 任务状态；如 PRD 或 ADR 变化，同步更新。
14. 最终汇报必须包含功能细节、测试证据、门禁状态、文档回写信息。

不能跳过测试直接实现；不能用“已手测”替代自动化测试；不能在相关测试红的状态下声称完成。

## 8. 测试纪律

采用 TDD + 测试金字塔。

### 8.1 验收标准先行

每个功能写测试前必须明确：正常路径、异常路径、边界条件、成功的可观察结果，以及对应 PRD/ROADMAP 或业务铁律 A#。

### 8.2 红绿重构

1. RED：先写失败测试，名称清楚表达目标行为。
2. Verify RED：运行确认它因功能缺失而失败。
3. GREEN：写最小实现让测试通过。
4. Verify GREEN：重跑，确认新测试和相关旧测试通过。
5. REFACTOR：仅在测试保持通过前提下重构。

测试第一次运行就通过，通常说明没有证明新行为，应重写或补充断言。

### 8.3 测试金字塔

- 单元测试：金额计算、统计聚合、分类删除/停用规则、账本删除限制、权限判断、图片校验、纯函数。
- 集成测试：Express API、Prisma 数据库读写、JWT 鉴权、上传接口、跨模块协作。
- E2E 测试：注册登录 -> 记账 -> 明细 -> 图表 的关键用户闭环；只覆盖真正用户旅程和高风险降级路径。

能用单元测试证明的业务规则不上 E2E；多模块协作优先集成测试；只有真实用户 journey 才用 E2E。

## 9. 提交门禁

工程初始化后必须提供可运行脚本。提交前本地至少通过：

- 前端：lint、typecheck、test、build。
- 后端：lint、typecheck、test。
- 涉及数据库：Prisma schema 校验、迁移生成或应用验证。
- 涉及关键闭环：对应集成测试或 E2E。

如果某项门禁暂不存在，最终汇报必须说明当前缺口、替代验证方式和补齐位置。不能把“当前没有脚本”当作跳过验证的理由。

## 10. Review 标准

每轮实现后必须自查：

- 是否仍符合 PRD 页面范围和核心闭环。
- 是否推进 ROADMAP 明确任务。
- 验收标准是否被自动化测试覆盖。
- 是否选对测试层级。
- 是否更新或固定了 API/schema/contract。
- 是否所有金额都避免 float。
- 是否所有数据访问都校验用户归属。
- 是否分类、账本、图片删除遵守 A3-A6。
- 是否暴露真实密钥、token、生产连接串。
- 是否需要回写 PRD 或 ROADMAP。

发现 bug：先补可复现失败测试，再修复，再 review，再重跑相关测试。

## 11. Git 规则

如果仓库已初始化 Git：

- `main`：稳定分支，唯一合并目标。
- 分支命名：`feat/<short-name>`、`fix/<short-name>`、`docs/<short-name>`、`test/<short-name>`、`chore/<short-name>`。
- 不得覆盖用户已有改动；只 stage 当前任务相关文件。
- commit 使用 Conventional Commits，例如：

```text
feat(api): add bill CRUD endpoints
fix(category): disable used categories instead of deleting
test(auth): cover jwt user isolation
docs(roadmap): mark phase 0 scaffold complete
```

当前若 Git 不可用，应在最终汇报说明。

## 12. PR / 汇报要求

PR 描述或最终汇报至少包含：

- Roadmap 来源：对应 Phase / R-###。
- PRD 对齐：相关 PRD 章节或业务铁律 A#。
- 变更类型：feat / fix / docs / test / chore。
- 变更细节：做了什么、关键文件、行为变化。
- 验收标准：正常路径、异常路径、边界条件、可观察结果。
- 测试策略：单元、集成、E2E 覆盖说明。
- 测试证据：运行命令与结果摘要，若没跑说明原因。
- Review 信息：自查过哪些风险。
- 文档回写：ROADMAP / PRD 是否更新；若未更新说明原因。
- 已知限制或后续任务。

## 13. 完成定义 DoD

只有同时满足以下条件才算完成：

- 验收标准明确并可追溯到 PRD/ROADMAP 或用户请求。
- 对应自动化测试先失败、实现后通过，测试层级合理。
- lint、typecheck、单元测试、集成测试通过；受影响 E2E 通过或说明无需 E2E。
- 相关 schema、迁移、API 契约更新并有测试。
- review 中发现的问题已修复并重跑相关测试。
- ROADMAP 对应任务进度已回写，或说明无需回写。
- PRD 已按产品/架构变化回写，或说明无需回写。
- 最终汇报包含测试证据、门禁结果和文档回写状态。

没有测试证据，不算完成。没有检查 PRD/ROADMAP 回写，不算完成。
